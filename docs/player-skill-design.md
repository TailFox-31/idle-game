# Player Skill System Design v1

## 문서 목적

이 문서는 현재 `main` 브랜치의 자동 전투 프로토타입 위에 플레이어 스킬 시스템을 얹기 위한 1차 설계안이다.
핵심 목표는 새 플레이어 스킬을 별도 전용 시스템으로 키우는 것이 아니라, 이미 보스에 들어간 메커닉 층을 이후 플레이어도 재사용할 수 있도록 방향을 고정하는 것이다.

지금 단계에서는 구현보다 설계가 목적이며, 다음 작업에서 MVP 범위를 바로 선택하고 착수할 수 있어야 한다.

## 왜 지금 스킬 시스템이 필요한가

현재 전투는 다음 요소로 이미 최소 루프가 성립해 있다.

- `AutoBattleSystem`이 플레이어와 적의 공격/피격/재생/리스폰을 단일 루프로 처리한다.
- `EnemyController`가 보스 웨이브마다 서로 다른 보스 메커닉을 부여한다.
- `BossMechanicRuntime`이 시간 기반 상태 전이와 피해 수정, 반사, 회복, 임계치 발동을 담당한다.

이 상태는 "적은 특수 규칙을 갖고, 플레이어는 기본 공격 스탯만 가진다"는 비대칭 구조다. 프로토타입 단계에서는 충분하지만, 다음 단계에서 아래 문제가 바로 생긴다.

- 전투 선택지가 업그레이드 수치 상승에만 묶여 있어 플레이 감각 분화가 약하다.
- 보스가 쓰는 패턴을 플레이어가 대응하거나 역이용하는 설계가 아직 없다.
- 이후 스테이지/보스 설계를 늘릴수록 보스 전용 예외 코드가 쌓일 가능성이 크다.

따라서 지금 스킬 시스템을 먼저 설계해야 하는 이유는 "새 콘텐츠 추가"보다 "메커닉 계층을 공통화할 시점"이기 때문이다. 지금 공통 기반을 잡아두면 이후 보스와 플레이어가 같은 effect/state 모델 위에서 움직일 수 있다.

## 현재 전투 루프와의 맞물림

현재 루프는 사실상 아래 순서로 동작한다.

1. 플레이어 재생 처리
2. 적 보스 메커닉 tick
3. 플레이어 공격 시도
4. 적 공격 시도
5. 적중 결과에 따른 사망/리스폰/보상/웨이브 전환 처리

여기서 플레이어 스킬은 아래 위치에 자연스럽게 들어간다.

- 전투 시작/리스폰 시 초기화
- 매 프레임 tick으로 쿨다운, 충전, 활성 상태 갱신
- 플레이어 공격 직전/직후의 피해 수정
- 플레이어 피격 직전/직후의 피해 감소, 반사, 임계치 발동
- UI용 상태 텍스트/남은 시간/사용 가능 여부 노출

즉, 플레이어 스킬은 새 전투 루프를 만드는 것이 아니라 현재 `BossMechanicRuntime`이 담당하는 역할을 플레이어 쪽에도 열어주는 형태가 가장 안전하다.

## 설계 원칙

- 보스 전용 메커닉을 플레이어 전용 로직으로 복제하지 않는다.
- "타이밍 창", "임계치 발동", "피해 증폭/감소", "반사", "자체 회복" 같은 효과를 공통 effect 단위로 재정의한다.
- MVP에서는 수동 발동 버튼이 필요한 스킬과 자동 발동형 스킬을 섞되, 모두 하나의 runtime 표현으로 관리한다.
- 초기 구현은 `AutoBattleSystem` 내부 확장으로 끝낼 수 있어야 하며, 씬/프리팹 대수정은 피한다.

## 제안 아키텍처 방향

### 1. 보스 메커닉을 공통 전투 메커닉으로 일반화

현 구조의 `BossMechanicDefinition`과 `BossMechanicRuntime`은 개념적으로 이미 재사용 가능하다. 다만 이름과 입력/출력 경로가 적 전용으로 잠겨 있다.

다음 단계 구현에서는 아래처럼 일반화하는 방향이 적절하다.

- `BossMechanicType` -> 전투 공통 `CombatMechanicType` 또는 `CombatEffectType`
- `BossMechanicDefinition` -> 플레이어/적 모두 쓸 수 있는 정의 데이터
- `BossMechanicRuntime` -> 발동 주체와 대상이 달라도 동일하게 tick 가능한 runtime

핵심은 "누가 쓰는가"보다 "언제 발동하고 무엇을 바꾸는가"를 분리하는 것이다.

### 2. 발동 모델은 두 갈래만 허용

초기 MVP에서는 발동 경로를 아래 둘로 제한한다.

- 수동 발동: 플레이어가 버튼으로 사용
- 자동 발동: 체력 임계치, 피격 이벤트, 반사 창 등 조건 충족 시 발동

이렇게 하면 입력/UI/세이브 부담을 제한하면서도 보스 메커닉 재사용성을 유지할 수 있다.

### 3. 플레이어 스킬은 "스탯 업그레이드"와 별도 층으로 둔다

현재 업그레이드는 영구 성장이고, 스킬은 전투 중 상태 변화를 만드는 능력이다. 둘을 한 시스템으로 섞으면 저장/UI/밸런싱 경계가 흐려진다.

권장 분리:

- 업그레이드: 영구 수치 성장
- 스킬: 전투 중 재사용 가능한 상태 전이

## MVP 범위 제안

초기 플레이어 스킬은 4종을 권장한다. 3종으로도 시작 가능하지만, 4종이 되어야 현재 보스 메커닉 재사용 의도가 명확하게 드러난다.

### 1. Burst

- 역할: 다음 1회 공격을 크게 강화하는 수동 공격 스킬
- 플레이 감각: 타이밍을 잡아 큰 피해를 넣고, 직후 취약 구간을 감수
- 재사용 대상: `WindUpBurst`

보스 재사용 포인트:

- 충전 상태 -> 준비 상태 -> 사용 후 회복 상태라는 3단계 전이
- 공격 직전 피해 배수 적용
- 사용 후 피격 취약도 증가

플레이어 적용 해석:

- 버튼 입력 시 즉시 "charge 시작" 또는 "ready 즉시 진입"
- 준비 상태에서 다음 기본 공격 1회가 강화
- 사용 후 짧은 취약 창을 둬서 무지성 연타를 막음

### 2. Frenzy

- 역할: 일정 시간 공격 속도 상승, 공격력 소폭 보정이 들어가는 수동 버프
- 플레이 감각: 짧은 폭딜 창
- 재사용 대상: `FrenzyWindow`

보스 재사용 포인트:

- 활성 시간과 비활성 시간의 교대
- 활성 중 공격 속도 배수
- 필요 시 공격력 배수 보정

플레이어 적용 해석:

- 사용 시 일정 시간 활성
- 활성 동안 평타 DPS 상승
- 자동 전투와 가장 잘 맞는 첫 버튼형 스킬

### 3. Guard

- 역할: 일정 시간 피해 감소 + 소량 회복의 생존 스킬
- 플레이 감각: 보스 폭딜/반사 창을 버티는 방어 선택지
- 재사용 대상: `GuardRecovery`

보스 재사용 포인트:

- 활성 창 동안 피해 감소
- 틱 기반 회복
- 활성 중 공격 불가 또는 공격 효율 저하라는 제약 가능

플레이어 적용 해석:

- 사용 시 피해 감소와 회복이 동시에 활성
- MVP에서는 공격 차단까지 넣지 않고, 공격 지속 여부는 밸런싱 결과로 결정
- 이후 난도가 올라가면 "Guard 중 공격 중단" 옵션을 다시 검토

### 4. Last Stand

- 역할: 체력이 특정 비율 이하로 내려가면 자동 발동하는 임계치 스킬
- 플레이 감각: 죽기 직전 버티기/역전 창
- 재사용 대상: `EnrageThreshold`

보스 재사용 포인트:

- 체력 임계치 검사
- 1회성 발동 플래그
- 발동 시 즉시 공격 템포/생존력 상승

플레이어 적용 해석:

- 체력 50% 이하 최초 진입 시 자동 발동
- 공격 속도 상승 또는 피해 감소 중 한 축을 선택
- MVP에서는 "전투당 1회" 규칙으로 제한

## 확장 후보

MVP 직후 가장 자연스러운 5번째 후보는 `Reflect`다.

### Reflect

- 역할: 짧은 반사 창 동안 받은 피해 일부를 적에게 되돌리는 수동/자동 방어 스킬
- 재사용 대상: `ReflectWindow`

보스 재사용 포인트:

- 활성 창 동안 반사
- 피해 감소와 반사 배율 결합
- 피격 이벤트 후 처리

보류 이유:

- 반사는 전투 로그/피드백이 약하면 체감이 낮다.
- 플레이어가 죽는 순간과 반사 적용 순서가 엮여, 초기 디버깅 난도가 다른 4종보다 높다.

따라서 1차 구현에서는 설계만 포함하고 실제 MVP 목록에서는 후보군 뒤쪽에 두는 편이 안전하다.

## 기존 보스 메커닉과 스킬 연결표

| 플레이어 스킬 | 재사용할 보스 메커닉 | 공유 상태 전이/효과 | 비고 |
| --- | --- | --- | --- |
| Burst | WindUpBurst | charge, ready, recovery, 다음 공격 강화, 취약 창 | 가장 직접적인 재사용 |
| Frenzy | FrenzyWindow | active/cooldown window, 공격 속도 배수, 공격력 보정 | 수동 발동형으로 전환 |
| Guard | GuardRecovery | active/cooldown window, 피해 감소, 틱 회복 | 공격 차단 여부는 후속 결정 |
| Last Stand | EnrageThreshold | 체력 임계치 감지, 1회 발동, 즉시 상태 변화 | 자동 발동형 |
| Reflect | ReflectWindow | 피격 시 반사, 피해 감소, active window | MVP 다음 후보 |

## 자원 모델 제안

MVP 기준 기본 자원 모델은 `개별 스킬 쿨다운 + 일부 스킬의 단발 활성 시간` 조합을 권장한다.

### 왜 쿨다운이 맞는가

- 현재 `BossMechanicRuntime`이 이미 시간 기반 타이머와 활성 창을 다루고 있어 구현 재사용성이 가장 높다.
- 자동 전투 프로토타입에서는 버튼 입력 빈도가 높지 않아도 되므로, 마나/에너지 같은 지속 소모 자원보다 해석이 단순하다.
- 저장 데이터 부담이 작다. 전투 외 영구 자원 회복 규칙을 먼저 만들 필요가 없다.
- UI도 "사용 가능 / 남은 시간 / 활성 중" 3상태만 우선 노출하면 된다.

### 충전/게이지는 왜 뒤로 미루는가

- 충전형은 저장 시 "남은 충전 수, 재충전 타이머"까지 들어가며 상태가 늘어난다.
- 게이지형은 적중/피격/시간 경과 등 게이지 획득 규칙이 별도 설계를 요구한다.
- 둘 다 가능하지만, 지금은 보스 메커닉 재사용 방향을 확정하는 것이 우선이다.

### 권장 세부안

- Burst: 쿨다운 + 준비 상태 1회 소비
- Frenzy: 쿨다운 + 활성 시간
- Guard: 쿨다운 + 활성 시간
- Last Stand: 전투당 1회 자동 발동, 웨이브 시작/리스폰 시 리셋
- Reflect: 쿨다운 + 짧은 활성 시간

## 저장 데이터 영향

현재 저장은 `GameManager.SaveData`에 골드, 웨이브, 마일스톤, 업그레이드 레벨만 들어간다.

1차 구현에서는 저장 범위를 최소화하는 것이 맞다.

### MVP에서 저장해야 하는 것

- 해금된 스킬 목록 또는 장착 슬롯 정보
- 스킬 레벨을 둘 경우 해당 레벨
- 자동 사용 설정이 들어가면 on/off 플래그

### MVP에서 저장하지 않아도 되는 것

- 전투 중 남은 쿨다운
- 현재 활성 상태
- 임계치 발동 여부
- 남은 지속 시간

이유:

- 현재 저장은 전투 진행을 장기 유지하는 세이브라기보다 진행도 저장에 가깝다.
- 웨이브 이동, 사망, 재시작 시 전투 상태가 리셋되어도 현재 게임 흐름과 충돌이 적다.
- 전투 중 runtime 상태까지 저장하면 설계 복잡도가 과도하게 증가한다.

## UI 영향

현재 UI는 상단 읽기 전용 전투 정보와 좌측 업그레이드 버튼 중심이다. 스킬 UI는 그 옆에 독립 패널로 들어가는 것이 자연스럽다.

### MVP UI 요구

- 스킬 버튼 3~4개
- 각 버튼의 상태 표시: 사용 가능, 쿨다운 중, 활성 중
- 자동 발동형 스킬은 버튼 대신 상태 라벨 또는 토글로 표현 가능
- 적 `EnemyStateLabel`과 유사하게 플레이어 상태 텍스트 1줄 추가 검토

### 권장 표시 정보

- 스킬명
- 남은 쿨다운
- 활성 남은 시간
- 자동 발동 여부

### UI 최소 변경 원칙

- 기존 전투 읽기 영역을 재사용해 플레이어 상태 문구를 1줄 추가
- 스킬 버튼은 업그레이드 패널과 분리된 소형 패널로 추가
- 씬 수정보다 `UIBinder`의 런타임 생성 패턴을 활용하는 방향이 안전

## 단계별 구현 순서

### Phase 1. 공통 메커닉 추출

- 보스 전용 이름을 공통 전투 메커닉 이름으로 정리
- 플레이어/적 모두 사용할 수 있는 definition/runtime 인터페이스 설계
- 현재 보스 동작이 깨지지 않도록 호환 레이어 유지

산출물:

- 공통 mechanic type/definition/runtime
- 기존 보스 메커닉의 회귀 확인

### Phase 2. 플레이어 스킬 슬롯/정의 계층 추가

- 플레이어가 보유한 스킬 정의 데이터 추가
- 수동 발동형과 자동 발동형 스킬 메타데이터 정리
- MVP 스킬 3~4종을 definition 수준에서 등록

산출물:

- 스킬 목록/슬롯 데이터 구조
- 초기 장착 스킬 세트

### Phase 3. 전투 루프 연결

- `AutoBattleSystem`에 플레이어 mechanic runtime tick 추가
- 플레이어 공격 전/후, 피격 전/후 후킹 지점 추가
- 스킬 상태를 `BattleSnapshot` 또는 별도 snapshot에 노출

산출물:

- Burst/Frenzy/Guard/Last Stand 동작
- 적 보스 메커닉과 플레이어 스킬 동시 동작 검증

## 밸런싱 포인트

- Burst는 현재 공격력 업그레이드와 곱연산으로 겹칠 때 과도한 순간 화력을 만들 수 있다.
- Frenzy는 공격속도 상승이 현재 자동 전투 루프에서 가장 체감이 강하므로, 지속 시간과 쿨다운 비율이 중요하다.
- Guard는 회복과 피해 감소를 동시에 주면 생존 효율이 급격히 뛸 수 있어 둘 중 하나의 수치를 낮게 잡아야 한다.
- Last Stand는 자동 발동이므로 체감은 좋지만, 강하면 사실상 추가 생명처럼 작동한다.
- Reflect는 고공격력 보스 상대로 지나치게 강해질 수 있어 반사량 상한 또는 피해 감소와의 동시 적용 규칙이 필요하다.

## 위험요소

### 1. 보스/플레이어 분기 코드가 다시 갈라질 위험

이름만 공통화하고 실제 내부 로직을 따로 두면 설계 목적이 사라진다. 공통 runtime과 훅 지점을 먼저 정리해야 한다.

### 2. 자동 전투에서 수동 스킬 조작이 과도하게 바쁠 위험

버튼형 스킬이 너무 많아지면 idle 성격이 약해진다. MVP는 수동 2~3개 이하가 적절하다.

### 3. 상태 텍스트 부족으로 체감이 흐려질 위험

특히 Reflect와 Guard는 발동 여부가 보이지 않으면 시스템이 있는지 체감하기 어렵다.

### 4. 저장 범위를 넓히다 구현이 비대해질 위험

전투 중 runtime 상태 저장은 1차 구현 목표가 아니다. 해금/장착 정도로 제한하는 편이 맞다.

## MVP 권장 결론

첫 구현 묶음은 아래 조합을 권장한다.

- Burst
- Frenzy
- Guard
- Last Stand

이 조합이면 현재 보스 메커닉의 네 가지 핵심 축을 그대로 플레이어 쪽으로 가져오면서도, 반사처럼 처리 순서가 까다로운 요소는 뒤로 미룰 수 있다.

## 구현 직전 결정해야 할 항목

- 플레이어 스킬 버튼 수: 3개로 제한할지, 4개까지 바로 열지
- Guard 중 공격 허용 여부
- Last Stand 효과를 공격형으로 둘지 생존형으로 둘지
- 스킬 해금/장착 시스템을 이번 단계에 포함할지, 고정 장착으로 시작할지
- 보스와 플레이어가 완전히 같은 definition/runtime를 공유할지, 공통 기반 + 얇은 어댑터로 갈지

위 다섯 항목만 확정되면 다음 단계 구현 범위는 바로 잘라낼 수 있다.
